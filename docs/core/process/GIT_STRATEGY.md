# ⚔️ 군단 깃 브랜치 전략: [Gajae Flow] v1.0

본 문서는 가재 컴퍼니의 코드 무결성을 수호하기 위한 최상위 브랜치 운영 표준입니다. 모든 가재 지능은 본 가이드를 1px의 오차 없이 준수해야 합니다.

---

## 🏛️ 브랜치 위계 구조 (Hierarchy)

### 1. `main` (The Sanctuary)
- **성격**: 무결성이 100% 검증된 '성역' 브랜치.
- **규칙**: 직접 커밋 및 푸시가 기술적으로 차단됨. 오직 `release` 브랜치로부터의 머지만 허용함.
- **상태**: 언제나 프로덕션 배포가 가능한 최적(OPTIMAL) 상태 유지.

### 2. `release` (The Testing Grounds)
- **성격**: 배포 전 최종 검수 및 통합 테스트가 이루어지는 구역.
- **규칙**: `feature` 브랜치들로부터의 작업본을 통합함.
- **D.O.D (Definition of Done)**:
    - 전수 테스트 패스 (QA 승인)
    - 헌법 및 리더십 원칙 무결성 검수 PASS
    - 1px 단위 UI/UX 정밀 검수 완료

### 3. `feature/*` (The Arena)
- **성격**: 개별 가재들의 연산과 코딩이 격돌하는 작업 구역.
- **명명 규칙**: `feature/가재ID-기능명` (예: `feature/DEV-api-optimization`)
- **규칙**: 기능 단위로 생성하며, 작업 완료 후 `release` 브랜치로 PR(Pull Request)을 요청함.

---

## 🔄 워크플로우 (Swarm Workflow)

1. **명령 수신**: CEO 명령 또는 시스템 트리거 발생.
2. **브랜치 생성**: 관련 가재가 `feature/*` 브랜치 생성 및 작업 개시.
3. **군집 리뷰**: 작업 완료 후 `release` 브랜치로 머지 요청 및 타 가재들의 교차 리뷰 수행.
4. **최종 승격**: `release` 구역의 모든 검증이 끝나면 `main` 성역으로 안치 및 프로덕션 배포 집행.
